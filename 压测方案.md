# AI 项目（MongoDB 升配）压测方案

## 1. 背景与目标

配合 DBA 对 MongoDB 分片集群进行内存升配（`wiredTigerCacheSizeGB=32`，实例升至 `r6i.2xlarge`）。在 DBA 逐分片关停从节点 → 升配 → 主从切换的窗口期内，由测试组持续向核心业务接口施加稳定负载，以：

1. 验证升配过程中业务可用性（无 5xx、无明显抖动）。
2. 暴露主从切换瞬间的连接抖动 / 慢查询 / 错误率突刺。
3. 配合研发、运维观测应用 / DB 错误日志，复核升配收益。

## 3. 被压接口与流量分配

总目标：50 并发线程。**投注事务（点水→投注 串联）占 50%**，其余 8 个接口均分剩余 50%（每个 6.25%）。

| # | 接口 | 方法 | 类别 | 流量占比 | 说明 |
| --- | --- | --- | --- | --- | --- |
| 1 | `/ai/mobile/bet/order/morePW` + `/ai/mobile/bet/order/moreBetting` | POST | 注单（投注事务） | **50%** | 串联事务：每次先点水拿实时赔率再投注 |
| 2 | `/ai/mobile/player/getBalance` | GET | 热门 | 6.25% | |
| 3 | `/ai/mobile/player/playerAccount` | GET | 热门 | 6.25% | |
| 4 | `/ai/mobile/home/common/matches` | GET | 热门 | 6.25% | |
| 5 | `/ai/mobile/dataPageServer/api/c/game/getGameManyInfo` | GET | 热门 | 6.25% | |
| 6 | `/ai/mobile/matches/getRBCondition` | GET | 滚球 | 6.25% | |
| 7 | `/ai/mobile/matches/commonMatches/orderLeagueList` | GET | 滚球 | 6.25% | |
| 8 | `/ai/mobile/player/capitalRecords` | GET | 钱包 | 6.25% | |
| 9 | `/ai/mobile/order/all/betRecordTab` | POST | 注单 | 6.25% | |

> 流量分配通过 JMeter `Throughput Controller`（百分比模式）实现；投注事务内的两个请求会被同一控制器命中，因此点水/投注的实际 TPS 接近 1:1。

### 投注参数化设计（动态链路）

为避免长时间用同一场比赛的固定赔率反复投注（不真实、易被风控、无法暴露真实 DB 查询路径），采用如下策略：

1. **setUp 阶段**：用首个登录用户调用 `/home/common/matches?gameSort=1&pageSize=50&groupId=1` 拉取热门赛事池，递归解析所有含 `gidm/gameId` 的对象，按 `gidm_gameId` 去重后写入 `props.BET_POOL`（全局共享）。
2. **运行期**：每次进入投注事务时，`JSR223 PreProcessor` 从赛事池**随机**抽取一场比赛，写入线程变量 `sel_gidm/sel_gameId/...`。
3. **点水（morePW）**：使用上述随机比赛信息发起请求，从响应中递归提取首条带 `ior` 的项，得到**实时赔率** `ior / ratio / ratioIpo / concede / score / betItem`，写入 `pw_*` 变量。
4. **投注（moreBetting）**：在 `If Controller`（`pw_ok=1`）保护下，使用上一步的实时赔率构造 body 提交。`gold` 固定为 50（页面限制 50~200，最小值降低风控概率）。若点水失败则跳过本次投注，避免脏数据。

这样做的好处：
- DB 查询命中范围从单条记录扩散到几十条赛事，能够更真实地暴露 Mongo 副本集的查询/缓存行为。
- 赔率每次取实时值，避免赔率漂移导致 100% 投注失败。
- 失败短路：点水失败 → 不投注，统计上更易区分上下游问题。

## 4. 压测策略（阶梯加压）

整体压测时长 **60 分钟**，前 10 分钟阶梯加压，后续维持 50 线程稳态运行。

| 阶段 | 持续时间 | 并发线程 | 说明 |
| --- | --- | --- | --- |
| Stage 1 | 0 ~ 2 min | 5 | 预热，验证脚本与登录态 |
| Stage 2 | 2 ~ 5 min | 15 | 观察基线 RT |
| Stage 3 | 5 ~ 8 min | 30 | 中等负载 |
| Stage 4 | 8 ~ 10 min | 50 | 目标负载 |
| Stage 5（稳态） | 10 ~ 60 min | 50 | 持续 50 分钟稳态压测，覆盖整个数据库变更窗口 |

> JMX 中通过 **内置 Thread Group + 长 ramp-up（600s）+ Scheduler（duration=3600s）** 实现等价的阶梯升压；如已安装 `jmeter-plugins-manager → Stepping Thread Group / Concurrency Thread Group`，可替换为更直观的阶梯模型。

### 流程编排
```
[setUp #1] 500 用户登录 → tokens.csv
[setUp #2] 用首个 token 拉热门赛事 → props.BET_POOL（全局共享）
        │
        ▼
[Main Thread Group]   50 线程 / 600s ramp-up / 3600s duration
   ├─ CSV: tokens.csv（loginName, token，循环复用）
   ├─ Header Manager（注入 token）
   └─ Loop：
       ├─ TC 50%   → 投注事务
       │             ├─ JSR223 PreProcessor: 随机选赛
       │             ├─ POST morePW（带回 ior/ratio/ratioIpo）
       │             └─ If pw_ok==1 → POST moreBetting (gold=50)
       ├─ TC 6.25% → getBalance
       ├─ TC 6.25% → playerAccount
       ├─ TC 6.25% → home/common/matches
       ├─ TC 6.25% → getGameManyInfo
       ├─ TC 6.25% → getRBCondition
       ├─ TC 6.25% → orderLeagueList
       ├─ TC 6.25% → capitalRecords
       ├─ TC 6.25% → betRecordTab
       └─ TC 6.25% → morePW（独立点水，覆盖空载赔率查询）
```

## 5. 监控与判定指标

### 客户端（JMeter）
- 总 TPS、各接口 TPS
- 平均 / P95 / P99 响应时间
- 错误率（< 0.1% 视为通过）
- 主从切换瞬间的 RT 突刺持续时长

### 服务端（研发 / 运维 / DBA）
| 维度 | 关注点 |
| --- | --- |
| 应用日志 | 5xx、超时、连接池打满、Mongo 连接异常 |
| MongoDB | `currentOp` 慢查询、副本集延迟（rs.printSecondaryReplicationInfo）、连接数、CPU/Mem |
| 中间件 | Redis、MQ 是否正常 |
| APM | 接口 P99 抖动、错误链路 |

### 阻断条件（任一触发立即停止压测）
- 投注接口连续 30s 错误率 > 5%
- 任一接口 P99 > 5s 持续 1 分钟
- 出现资金类异常（账务不平、扣款失败但订单成功等）

## 6. 时间线 & 协作

| 时间点 | 动作 | 负责方 |
| --- | --- | --- |
| T-30min | 启动 setUp 完成 token 预生成 | 测试 |
| T-10min | 启动主线程组（5→50 阶梯加压） | 测试 |
| T0 | DBA 开始第一个分片从节点升配 | DBA |
| T 全程 | 观察日志 / DB / APM | 研发、运维、DBA |
| T 每个分片完成 | 测试组短暂 5 分钟生产回归 | 测试 |
| T+60min | 停止压测，导出报告 | 测试 |

## 7. 输出物
- `AI压测.jmx`（可在 GUI 调试，CLI 跑压）
- JMeter 聚合报告（HTML）：`jmeter -n -t AI压测.jmx -l result.jtl -e -o report/`
- 错误样本 / 慢请求样本

## 8. 风险与回滚
- 压测账号与生产账号已隔离，但 `moreBetting` 真实落单，结束后由研发统一清理 `press001~press500` 注单。
- 若主从切换期间出现严重抖动，立刻 **降并发到 5 线程** 或停压；不阻断 DBA 操作流。

## 9. 执行命令示例

```bash
# 命令行执行（Linux/Mac/Windows 通用）
jmeter -n -t AI压测.jmx -l result.jtl -e -o report \
  -Jthreads=50 -Jrampup=600 -Jduration=3600 -Jusers=500
```

可调参数：

| 参数 | 默认 | 说明 |
| --- | --- | --- |
| `threads` | 50 | 主线程组并发数 |
| `rampup` | 600 | 主线程组爬升秒数（用于阶梯效果） |
| `duration` | 3600 | 主线程组运行秒数 |
| `users` | 500 | setUp 阶段登录的账号数（≤ user.csv 行数） |

JMX 内置全局变量：

| 变量 | 默认 | 说明 |
| --- | --- | --- |
| `BET_GOLD` | 50 | 投注金额（页面限制 50~200，固定取最小值） |
| `MERCHANT_ID` | 136ebd8d... | 商户 ID |
| `PASSWORD_MD5` | 62c8ad0a... | 压测账号统一密码 MD5 |
