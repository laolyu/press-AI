# AI 项目（MongoDB 升配）压测方案

## 1. 背景与目标

配合 DBA 对 MongoDB 分片集群进行内存升配（`wiredTigerCacheSizeGB=32`，实例升至 `r6i.2xlarge`）。在 DBA 逐分片关停从节点 → 升配 → 主从切换的窗口期内，由测试组持续向核心业务接口施加稳定负载，以：

1. 验证升配过程中业务可用性（无 5xx、无明显抖动）。
2. 暴露主从切换瞬间的连接抖动 / 慢查询 / 错误率突刺。
3. 配合研发、运维观测应用 / DB 错误日志，复核升配收益。

## 3. 被压接口与流量分配

总目标：50 并发线程。**投注事务（点水→投注 串联）占 50%**，其余 8 个接口均分剩余 50%（每个 6.25%）。

| # | 接口 | 方法 | 类别 | 流量占比 | 说明 |
| --- | --- | --- | --- | --- | --- |
| 1 | `/ai/mobile/bet/order/morePW` + `/ai/mobile/bet/order/moreBetting` | POST | 注单（投注事务） | **50%** | 串联事务：每次先点水拿实时赔率再投注 |
| 2 | `/ai/mobile/player/getBalance` | GET | 热门 | 6.25% | |
| 3 | `/ai/mobile/player/playerAccount` | GET | 热门 | 6.25% | |
| 4 | `/ai/mobile/home/common/matches` | GET | 热门 | 6.25% | |
| 5 | `/ai/mobile/dataPageServer/api/c/game/getGameManyInfo` | GET | 热门 | 6.25% | |
| 6 | `/ai/mobile/matches/getRBCondition` | GET | 滚球 | 6.25% | |
| 7 | `/ai/mobile/matches/commonMatches/orderLeagueList` | GET | 滚球 | 6.25% | |
| 8 | `/ai/mobile/player/capitalRecords` | GET | 钱包 | 6.25% | |
| 9 | `/ai/mobile/order/all/betRecordTab` | POST | 注单 | 6.25% | |

> 流量分配通过 JMeter `Throughput Controller`（百分比模式）实现；投注事务内的两个请求会被同一控制器命中，因此点水/投注的实际 TPS 接近 1:1。

### 投注参数化设计（动态链路）

为避免长时间用同一场比赛的固定赔率反复投注（不真实、易被风控、无法暴露真实 DB 查询路径），采用如下策略：

1. **setUp 阶段**：用首个登录用户调用 `/home/common/matches?gameSort=1&pageSize=50&groupId=1` 拉取热门赛事池，直接遍历 `data.games[]`，**只保留** `gameType==FT`（足球）**且存在 `M.game` 主赛盘口** 的赛事，取 `gidm/gameId/systemId/homeTeamSuffix/strong/leagueId` 入池，按 `gidm` 去重后写入 `props.BET_POOL`（全局共享）。BK（篮球）赛事的 `gameId` 字段含有 `systemId` 前缀，与 morePW 接口契约不一致，已过滤。
2. **运行期**：每次进入投注事务时，**挂在 morePW 采样器子节点的 JSR223 PreProcessor** 仅在 morePW 请求前随机抽一场比赛（`sel_gidm/sel_gameId/...`）。该 PreProcessor **不能**挂在 ThroughputController 子节点，否则在 moreBetting 前会被重复执行，导致点水/投注不是同一场比赛。
3. **点水（morePW）**：使用上述随机比赛信息发起请求，从响应 `data.orderData[0]` 直接提取**实时赔率** `ior / ratio / ratioIpo / concede / score / betItem / playType / ratioType`，写入 `pw_*` 变量；`errorCode != null` 或 `code != 200` 则跳过。
4. **投注（moreBetting）**：在 `If Controller`（`pw_ok=1`）保护下，使用上一步的实时赔率构造 body 提交。`playType/ratioType` 也动态取自 morePW 响应（避免硬编码 M/MH 与返回赔率不一致）。`gold` 取 `BET_GOLD` 变量（页面限制 50~200，取最小值降低风控概率）。

这样做的好处：
- DB 查询命中范围从单条记录扩散到几十条赛事，能够更真实地暴露 Mongo 副本集的查询/缓存行为。
- 赔率每次取实时值，避免赔率漂移导致 100% 投注失败。
- 失败短路：点水失败 → 不投注，统计上更易区分上下游问题。

## 4. 压测策略（四场景）

整体时长约 **90 分钟**，按顺序执行四个场景。JMX 使用 `TestPlan.serialize_threadgroups=true`，逐线程组串行。

| 场景 | 时长 | 并发 / TPS | 目的 |
| --- | --- | --- | --- |
| 场景一：基准测试（扩容前） | 10 min | 10 线程（约 50% 额定） | 获取扩容前基线 RT；验证脚本/参数正确 |
| 场景二：满载测试（扩容前） | 15 min | 50 线程（100% 额定），60s ramp-up | 记录扩容前 MongoDB 满载性能基线（CPU/内存/延迟） |
| 场景三：Rolling Upgrade 观察 | ~45 min | 50 线程持续施压，全程不停 | **核心场景**：观察升配切换期间的抖动、恢复时间 |
| 场景四：满载测试（扩容后） | 15 min | 50 线程（与场景二相同参数） | 验证升配后性能是否提升，与场景二基线对比 |

### 场景一：基准测试（扩容前，10 分钟）
- 线程数：**10**，ramp-up 30s，duration 600s
- 目的：验证功能正确性与参数有效性，获取低压下的响应时间基线
- 通过标准：错误率 = 0，所有接口 P95 < 1s

### 场景二：满载测试 – 扩容前（15 分钟）
- 线程数：**50**，ramp-up 60s，duration 900s
- 目的：记录扩容前 MongoDB 在满载下的性能基线（CPU / 内存 / 延迟）
- **在此停下，等待 DBA 开始升配**

### 场景三：Rolling Upgrade 观察（核心，约 45 分钟）
- 线程数：**50**，ramp-up 0s，duration 2700s（升配前 5min + DBA 执行 ~30min + 升配后 10min）
- 全程不停压，DBA 告知每次主从切换的精确时刻
- **关键观测点**：
  - 主从切换前后各 1 分钟的错误率变化
  - 切换期间出现的错误类型（连接超时 / 5xx / `code≠0`）
  - **服务自动恢复时间**：首个错误 → 错误率归零的耗时
  - 切换前后响应时间是否抖动（P99 突刺）

### 场景四：满载测试 – 扩容后（15 分钟）
- 线程数：**50**，ramp-up 60s，duration 900s（与场景二完全相同参数）
- 目的：与场景二基线对比，验证升配（r6i.2xlarge / 32G cache）性能提升

### JMX 参数化执行（四场景 CLI 示例）

```bash

jmeter -n -t AI压测.jmx -l s3_rolling.jtl -e -o report_s3 \

# 场景四：满载（扩容后）
jmeter -n -t AI压测.jmx -l s4_after.jtl -e -o report_s4 \
  -Jthreads=50 -Jrampup=60 -Jduration=900 -Jusers=500
```

### 各场景通过标准与对比维度

| 指标 | 场景一 基线 | 场景二 满载前 | 场景三 切换 | 场景四 满载后 |
| --- | --- | --- | --- | --- |
| 错误率 | = 0 | < 0.1% | 切换瞬间 < 5%，恢复后 = 0 | < 0.1% |
| 投注 P95 RT | < 1s | < 2s | 切换期间允许抖动 | 应优于场景二 |
| 投注 P99 RT | < 2s | < 3s | 恢复后应归零 | 应优于场景二 |
| 服务恢复时长 | N/A | N/A | **< 60s（目标）** | N/A |

### 场景三 时间线标注建议
- 请 DBA 在每次执行 `rs.stepDown()` 和服务重启完毕时，**在钉钉/群里报时间戳**
- JMeter 结合 `.jtl` 文件时间戳，即可精确定位抖动区间

### 流程编排
```
[setUp #1] 500 用户登录 → tokens.csv
[setUp #2] 用首个 token 拉热门赛事 → props.BET_POOL（全局共享）
        │
        ▼
[Main Thread Group]   50 线程 / 600s ramp-up / 3600s duration
   ├─ CSV: tokens.csv（loginName, token，循环复用）
   ├─ Header Manager（注入 token）
   └─ Loop：
       ├─ TC 50%   → 投注事务
       │             ├─ POST morePW（点水）
       │             │     ├─ JSR223 PreProcessor：随机选赛（**挂在 morePW 子节点**）
       │             │     └─ JSR223 PostProcessor：解析 orderData[0] 得实时赔率
       │             └─ If pw_ok==1 → POST moreBetting (gold=BET_GOLD)
       ├─ TC 6.25% → getBalance
       ├─ TC 6.25% → playerAccount
       ├─ TC 6.25% → home/common/matches
       ├─ TC 6.25% → getGameManyInfo
       ├─ TC 6.25% → getRBCondition
       ├─ TC 6.25% → orderLeagueList
       ├─ TC 6.25% → capitalRecords
       ├─ TC 6.25% → betRecordTab
       └─ TC 6.25% → morePW（独立点水，覆盖空载赔率查询）
```

## 5. 监控与判定指标

### 客户端（JMeter）
- 总 TPS、各接口 TPS
- 平均 / P95 / P99 响应时间
- 错误率（< 0.1% 视为通过）
- 主从切换瞬间的 RT 突刺持续时长

### 服务端（研发 / 运维 / DBA）
| 维度 | 关注点 |
| --- | --- |
| 应用日志 | 5xx、超时、连接池打满、Mongo 连接异常 |
| MongoDB | `currentOp` 慢查询、副本集延迟（rs.printSecondaryReplicationInfo）、连接数、CPU/Mem |
| 中间件 | Redis、MQ 是否正常 |
| APM | 接口 P99 抖动、错误链路 |

### 阻断条件（任一触发立即停止压测）
- 投注接口连续 30s 错误率 > 5%
- 任一接口 P99 > 5s 持续 1 分钟
- 出现资金类异常（账务不平、扣款失败但订单成功等）

| T+25min~T+70min | 研发 / 运维观察日志 / DB / APM | 研发、运维 |
| T+70min | 场景三结束，等待 5 分钟生产回归 | 测试 |

## 7. 输出物
- 四份 JMeter HTML 报告（`report_s1 ~ report_s4`）及对应 `.jtl` 文件
- 场景二 vs 场景四 对比表（TPS / P95 / P99 / 错误率）
- 场景三时间轴：标注各分片切换时刻、首个错误时刻、恢复时刻、持续时长
- 错误样本 / 慢请求样本

## 8. 风险与回滚
- 压测账号与生产账号已隔离，但 `moreBetting` 真实落单，结束后由研发统一清理 `press001~press500` 注单。
- 若主从切换期间出现严重抖动，立刻 **降并发到 5 线程** 或停压；不阻断 DBA 操作流。

## 9. 执行命令示例
